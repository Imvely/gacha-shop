# /done — 완료 판정 게이트

대상 기능: **$ARGUMENTS**

features.json의 status를 `done`으로 바꾸는 것은 **오직 이 커맨드를 통해서만** 한다.
이 게이트의 목적은 "self-grading bias" 방지 — 에이전트는 자기 작업을 후하게 평가하는
경향이 있으므로, 완료 판정은 선언이 아니라 증명이어야 한다.

## 절대 규칙

- **acceptance 배열을 수정·삭제·완화해서 통과시키는 것은 금지.** 기준이 잘못됐다고
  판단되면 status를 건드리지 말고 사용자에게 기준 변경을 제안하라.
- 테스트를 지우거나 skip 처리해서 통과시키는 것도 동일하게 금지.

## 수행 순서

1. features.json에서 해당 기능의 acceptance 배열을 읽고 체크리스트로 나열하라.

2. **항목별 실제 검증**: 각 항목을 코드 읽기가 아니라 실행으로 증명하라.
   - 백엔드: 해당 엔드포인트를 실제로 호출 (pytest 통합 테스트 또는 로컬 서버 + 실요청)
   - 프론트: 실제 렌더링 확인 (테스트 또는 브라우저 구동), 375px 항목은 해당 뷰포트로
   - 돈/재고 기능(F-02, F-04, F-09, F-11 등): **pytest 스레드 20개 동시 draw 테스트 +
     원장 불변식 테스트 통과 필수** (money-safety 스킬의 게이트 체크리스트 참조)

3. **판정**:
   - 전 항목 통과 → features.json status를 `done`으로 갱신,
     `[F-xx] done: <기능명>` 커밋. 검증 증거(돌린 명령과 결과 요약)를 커밋 메시지 본문에 남겨라.
   - 하나라도 실패 → status는 `in_progress` 유지. 실패 항목과 원인을 STATE.md
     "막힌 것"에 기록하고, 실패 내용을 그대로 보고하라. 통과한 척 금지.

4. **결과 보고**: 항목별 ✅/❌ 표와 각 항목의 증거(실행한 명령, 핵심 출력)를 보여라.
